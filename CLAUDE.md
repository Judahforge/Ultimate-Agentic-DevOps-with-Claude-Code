# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Static HTML/CSS portfolio website (the "DMI Portfolio Website"). Pure HTML5 + CSS3, **no JavaScript and no build step** — files are served as-is. Used in the DevOps Micro Internship (DMI) to practice hosting and deployment.

There are two deployment stories in this repo, and they target different infrastructure — do not conflate them:
- **Live CI/CD (`.github/workflows/deploy.yml`)** deploys to AWS S3 + CloudFront.
- **The README assignment** describes hosting manually on an Ubuntu VM with Nginx (`http://<public-ip>`). This is instructional and is not what the CI pipeline does.

## Layout

- `index.html` — single-page portfolio (About, Services, Courses, Books, Community, Contact)
- `style.css` — all styling; mobile-first responsive
- `privacy.html` / `terms.html` — standalone pages with inline styles
- `images/` — static assets (logo, profile, course thumbnails, hero)
- `.github/workflows/deploy.yml` — CI deploy pipeline
- `.claude/skills/` — custom skills for provisioning/deploying infra (see below)

Note: `terraform/` does not exist yet — it is generated by the `scaffold-terraform` skill.

## Deployment (CI/CD)

`deploy.yml` runs on every push to `main`:
1. Authenticates to AWS via **OIDC** (IAM role `github-actions-deploy`, account `533267262133`, region `eu-north-1`) — no stored access keys.
2. `aws s3 sync` to bucket `pravinmishradmi-site-production` with `--delete`, excluding `.git`, `.github`, `.claude`, `terraform`, `.mcp.json`, `*.md`.
3. Invalidates CloudFront distribution `E3V6O6MRE2E21P` (`--paths "/*"`).

The role ARN, bucket, and distribution ID are hardcoded to the original author's AWS account. Forking this repo to a different account requires updating lines 24, 29, and 42 of `deploy.yml`.

## Skills — infra & deploy work goes through these

All Terraform and deploy actions are handled by skills in `.claude/skills/`. All four have `disable-model-invocation: true`, so they run **only when the user explicitly invokes them** (`/scaffold-terraform`, `/tf-plan`, `/tf-apply`, `/deploy`) — do not trigger them automatically.

- `/scaffold-terraform [region] [name]` — generates the full `terraform/` config (S3 + CloudFront + OAC). Reads `.claude/skills/scaffold-terraform/template-spec.md` for the exact spec. Defaults: region `ap-south-1`, name `portfolio-site`.
- `/tf-plan` — `terraform plan -no-color` + risk/blast-radius analysis. On failure, diagnose; do not apply.
- `/tf-apply` — `terraform apply -auto-approve -no-color`, then verify CloudFront reaches "Deployed". **Never auto-retry on failure** — report and wait.
- `/deploy` — `aws s3 sync` + CloudFront invalidation, reading bucket/distribution from `terraform output -json`. Each step gates the next: if a step fails, stop and report.

When generating Terraform, follow `template-spec.md` exactly (private S3, CloudFront OAC — not legacy OAI, `redirect-to-https`, 404→/index.html, `PriceClass_200`, `Project`/`Environment` tags, commented-out S3 backend with migrate-state instructions).

## Commands

```bash
# Local preview — just open the file, there is no dev server
start index.html          # Windows

# Terraform (only after /scaffold-terraform has generated terraform/)
cd terraform && terraform init
cd terraform && terraform plan
cd terraform && terraform apply
```
